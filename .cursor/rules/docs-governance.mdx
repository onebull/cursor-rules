# 文档治理与生命周期规范
> **核心原则**：最小化创建、优先修改追加、临时文档即用即删、日期必须实时获取。
---
## 一、时间戳与实时性
- **强制获取 UTC 时间**：在文档中使用任何日期之前，必须先执行 `date` 命令获取系统当前时间（UTC 时区）。
- **禁止猜测日期**：严禁基于训练数据或记忆中的日期编写文档时间戳。
- **统一格式**：`YYYY-MM-DD`（如 `2025-12-27`）。
- **时区标准**：始终使用 **UTC**。
---
## 二、文档创建原则（最小化）
### 2.1 讨论阶段策略（混合模式）
| 任务类型 | 默认处理 |
|----------|----------|
| 简单任务（单轮即可确认，影响范围小） | 在对话中完成，**不创建文档** |
| 复杂任务（多轮讨论、需求不明确、影响大） | 可创建讨论文档 `work/YYYY-MM-DD-任务名-讨论.md` |
讨论文档**生命周期**：方案确认后立即删除，或归档到 `docs/archive/`（仅限有长期参考价值时）。
### 2.2 文档创建时机
1. **讨论阶段**：默认不创建，仅复杂任务例外。
2. **方案阶段**：确认后创建方案文档（`docs/specs/` 或 `work/`），包含问题分析、方案设计、实施计划。
3. **执行阶段**：执行完成后创建总结文档（`work/YYYY-MM-DD-任务名.md`），包含完成工作、技术要点、验证步骤。
### 2.3 优先修改而非新建
- 创建文档前，必须检索 `docs/` 下是否存在相关规范、架构设计或分析文档。
- 若存在，应在原文档基础上**修改或追加**，而非新建。
- 同一任务或同类问题，**只能存在一个**分析文档，避免碎片化。
---
## 三、临时文档管理
- **即用即删**：所有临时分析报告、调试记录、任务草稿，在使用完成或任务确认后**立即删除**。
- **归档例外**：只有具备长期参考价值（如核心逻辑推演、重要决策记录）才允许归档到 `docs/archive/`。
- **每个 Turn 结束前**：检查并清理本次对话中产生的不再需要的临时 `.md` 文件。
---
## 四、文档目录约定（推荐结构）
```
docs/
├── architecture/ # 系统架构、服务设计
├── api/ # API 汇总、接口规范
├── runbooks/ # 启动指南、运维手册、操作手册
├── specs/ # 技术规范、一致性分析、设计方案（定稿）
└── ref/ # 参考资料、配置说明、工具使用指南
work/ # 工作日志与 AI 任务总结（历史档案）
├── YYYY-MM-DD.md # 日常工作日志
├── YYYY-MM-DD-任务名.md # AI 任务完成总结（必需）
└── YYYY-MM-DD-任务名-讨论.md # 讨论文档（确认后删除）
```
### 放置原则速查
| 文档类型 | 位置 |
|----------|------|
| 任务完成总结 | `work/YYYY-MM-DD-任务名.md` |
| 日常工作日志 | `work/YYYY-MM-DD.md` |
| 技术规范（定稿） | `docs/specs/` |
| 架构设计 | `docs/architecture/` |
| API 文档（全局） | `docs/api/` |
| 运维/启动手册 | `docs/runbooks/` |
| 参考资料、配置说明 | `docs/ref/` |
| 临时分析/草稿 | `work/`（完成后清理） |
---
## 五、任务总结文档规范（必填）
每个任务完成后必须创建总结文档，内容至少包含：
```markdown
# [任务名称]
**日期**：YYYY-MM-DD（UTC）
**任务**：[一句话描述]
## 任务目标
[背景与目标]
## 完成的工作
1. ✅ [工作项 1]
2. ✅ [工作项 2]
## 技术要点与关键决策
- [要点 1]
- [要点 2]
## 相关文件
- `path/to/file` — [说明]
## 验证步骤
```bash
# 最小验证命令
```
## 后续建议（如有）
- [建议]
```

## 六、执行阶段检查（质量清单）

执行阶段或任务交付前，应自检：

- [ ] 是否已通过 `date` 获取最新系统日期（UTC）？
- [ ] 本次修改是否可以在现有文档中完成（避免重复新建）？
- [ ] 是否已创建总结文档（执行后）？
- [ ] **新建或移动的文档是否已同步更新 `docs/INDEX.md`**（若项目中存在该文件且该文档应被索引）？
- [ ] 任务完成后，临时产生的草稿文档是否已清理？
- [ ] 问题分析是否已汇总到对应的单一文档中？

---
## 七、工作日志使用门禁
- `work/**` 仅为**历史参考/结案档案**，不得作为「现行规则/唯一依据」。
- 任何最终结论须与**现行规则源**（`.cursorrules`、`docs/specs/`）对齐；`work/` 与之冲突时以现行规则源为准。
- 引用 `work/**` 时须显式标注「**历史记录，可能过期**」。
---
## 八、项目扩展提示
> 以下为可选扩展方向，项目可按需在此文件追加规则或新建子规则文件。
```
# 可扩展维度（按需选用）
#
# 子项目文档目录
# - {子项目}/docs/api/ 子项目 API 文档
# - {子项目}/docs/deployment/ 部署/运维配置
# - {子项目}/docs/migration/ 数据库迁移文档
# - {子项目}/docs/specs/ 子项目技术规范
#
# 文档归档策略
# - 任务总结保留期限（如 2 个月）
# - 日常日志保留期限（如 1 个月）
# - 定期清理命令：find work/ -name "*.md" -mtime +60 -delete
#
# 文档索引
# - docs/INDEX.md 全局文档导航
# - docs/QUICK_REFERENCE.md 快速参考
# - 各子项目根 README.md
#
# 变更记录
# - CHANGELOG.md / FEATURE_LOG.md 功能变更记录
# - 按子项目维护 or 全局维护
```
