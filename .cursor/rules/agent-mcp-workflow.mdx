# Agent MCP 与最优流程（自动选用）

> **读者**：本规则供 Agent 自动选用 MCP、产出可执行清单与可落档产物。
> **目标**：所有中间产物**可见、可审核、可验收**，落档到固定路径，便于在仓库内打开、评审、按验收标准勾选。

---

## 一、自动选用原则

当出现以下情况时，**无需用户提醒**，自动选用对应工具：

| 场景 | 自动使用 | 目的 |
|------|----------|------|
| 任务需要**分析、拆解、多步推理** | Sequential Thinking MCP | 分步思考、可修订、可分叉，避免一步到位出错 |
| 需要**拆成可执行清单并持久化** | Tasks MCP | 任务写入 `.cursor/tasks.md`，跨会话可查、可勾选 |
| 需求来自文档/口头/竞品，需明确「做什么、验收标准」 | 先出 PRD/需求要点 | 背景与目标、范围、验收标准、约束与假设 |
| 涉及多模块、接口、与现有架构关系 | 先出方案/设计要点 | 关键决策、模块/接口、与现有代码关系、风险与备选 |

---

## 二、先产出 PRD/设计要点再拆任务（强制）

在**拆成可执行任务之前**，必须先产出**至少一种**中间产物，落档后再拆任务。

### 2.1 何时产出哪种

| 需求类型 | 必须产出 |
|----------|----------|
| 需求来自文档/口头，需明确「做什么、为什么、验收标准」 | **PRD/需求要点** |
| 涉及多模块、接口、与现有架构关系 | **方案/设计要点** |
| 同时满足「需求不清晰」且「多模块」 | 先 PRD 再设计要点，或合并为一篇 |
| 简单单文件任务 | 可在对话中完成，不强制落档 |

每种产物至少包含**验收标准**或「Done 条件」（一句话）。

### 2.2 任务粒度

- **单条任务** = 一个可独立验证的交付单元（1 个文件/1 个接口/1 个 UI 区块，或 1 个「可单独跑测或肉眼确认」的改动集合）
- **禁止**：一条任务包含多个无关文件或多个互不相关的功能点
- 每条任务建议带**验收要点 / Done 条件**（一句话），便于用户按条勾选验收

### 2.3 产物默认落档路径

> 日期用 `date -u` 获取，格式 `YYYY-MM-DD`。路径以项目根为基准，按项目约定调整。

| 产物 | 落档路径（默认） | 用户用途 |
|------|-----------------|----------|
| PRD/需求要点 | `work/YYYY-MM-DD-任务名-PRD.md` | 审核范围与验收标准 |
| 方案/设计要点 | `work/YYYY-MM-DD-任务名-设计要点.md` | 审核决策与接口 |
| 任务清单 | `.cursor/tasks.md`（Tasks MCP 持久化） | 查看进度、按条验收 |
| 竞品对照结果 | `work/YYYY-MM-DD-竞品对照-功能名.md` | 审核差距与建议 |

- 对话中可先给出摘要或关键结论，但**完整内容必须写入上述路径**，不得仅存在于对话中。

### 2.4 再拆任务

在 PRD/设计要点**落档并经用户确认**（或明确无异议）后，再用 Sequential Thinking 拆解、用 Tasks MCP 落清单（含验收要点）。

---

## 三、MCP 工具路径

- **Sequential Thinking**：用于复杂问题分步推理（MCP server 名称以项目配置为准）。
- **Tasks MCP**：任务文件路径 **`.cursor/tasks.md`**。首次使用前调用 `tasks_setup` 指定该路径；拆解出的任务用 `tasks_add` 写入，落实时用 `tasks_update` 更新状态。

---

## 四、与现有规则的关系

- **先方案后执行**：使用 Sequential Thinking 做方案分析时，仍遵守「方案→确认→执行」。
- **Memory Bank**：任务拆解落入 tasks.md 后，同步更新 activeContext.md（见 `memory-bank-workflow.mdx`）。
- **文档治理**：PRD/设计要点落档策略与 `docs-governance.mdx` 保持一致。

## 五、项目扩展提示

> 以下为可选扩展方向，项目可在此文件追加或新建 Skill 文件。

```
# 可扩展的 Skill / MCP 场景（按需添加）
#
# product-dev-workflow
#   - 标准工程流程：阶段 0（需求澄清）→ PRD → 设计要点 → 拆任务 → 落实
#   - 适用：用户说「帮我做 XX」「按固定流程开发」
#
# task-chain-from-doc
#   - 文档/任务 → PRD/设计要点 → 分析拆解 → 可执行清单 → 按项落实
#
# doc-and-feature-compare
#   - 参考某站做同样功能 → 抓取/对照 → 输出要补的清单
#   - 落档路径：work/YYYY-MM-DD-竞品对照-功能名.md
#
# browser-use / animation-check
#   - 用 Browser MCP 打开预览/URL → 截图 → 描述性判断与优化建议
#
# 自定义测试 Skill
#   - 用 Browser MCP 或终端 E2E 执行项目特定流程
#   - 失败则改代码重跑直至通过或重试上限
#
# external-knowledge-fetch（如 EvoMap、Context7 等）
#   - 实现新能力 / 修某类 bug 前，先拉取外部经验 Capsule 再设计方案
#
# Skills 变更须同步更新新机器复现指南（如 docs/runbooks/CURSOR_SKILLS_MCP_SETUP.md）
```
