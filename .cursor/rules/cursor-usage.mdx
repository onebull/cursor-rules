# Cursor 使用约定

> **适用范围**：所有在 Cursor 中进行的对话与 Agent 操作。  
> **目的**：统一 Composer/Chat、Plan、@ 引用与对话边界，提高准确率、减少无效上下文。

---

## 一、Composer 与 Chat 的选用

| 场景 | 推荐 | 说明 |
|------|------|------|
| 多文件修改、跨模块重构、新功能实现、涉及 3 个以上文件 | **Composer**（`Ctrl/Cmd+I`） | 多文件编辑核心入口，方案/任务拆解后在此执行 |
| 单点问答、理解某段代码、小范围改写、不落档的讨论 | **Chat**（`Ctrl/Cmd+L`） | 仅建议不直接改代码时用 Chat；要改代码且跨文件仍用 Composer |
| 行内生成/改写（单文件、单块） | **Inline**（`Ctrl/Cmd+K`） | 空行生成、选中改写；不改动其他文件时用 |

- 与根 `.cursorrules` 的「方案优先、执行确认」一致：在 Composer 中先给出方案或按 Plan 执行，再动手改代码。

---

## 二、Plan 模式（Shift+Tab）

- **何时用**：架构级改动、多文件/多模块、需求尚不清晰、需先列步骤再实现时，在 Agent 输入框按 **Shift+Tab** 进入 Plan 模式。
- **效果**：Agent 先检索代码库、提问澄清、产出可审查的 Markdown 计划（涉及文件与步骤），待用户确认后再执行。
- **与方案讨论的关系**：根规则的「询问句→方案讨论」偏产品/方案层；Plan 模式偏「执行计划」层。复杂任务可先方案讨论再在 Composer 中开 Plan 模式出实施计划。
- **计划落点**：优先 `work/YYYY-MM-DD-任务名-计划.md`（与文档治理一致）；若使用 Cursor 内置「Save plan」存至 `.cursor/plans/` 也可接受。详见 `agent-mcp-workflow.mdx` 产物落档路径。

---

## 三、@ 引用

- **能明确文件/目录时**：用 `@文件名`、`@文件夹` 精确引用，减少无关文件进上下文。
- **不确定改哪时**：用 `@Codebase` 做语义搜索，让 Agent 自己定位。
- **需依据官方文档时**：用 `@Docs`（语言/框架文档）。
- **看最近改动时**：用 `@Git`。
- **需要旧对话结论时**：用 `@Past Chats`，避免长对话里复制粘贴。
- **避免**：一次性 @ 过多无关文件；不确定时优先 @Codebase 或限定子目录。

---

## 四、部分接受与编辑习惯

> **仅供人类用户**：以下键盘快捷键为 Cursor IDE 操作，Agent 无法触发，此节供用户参考。

- **部分接受建议**：使用 **Ctrl/Cmd+右箭头** 逐段接受 AI 补全，不必整块接受再删除不需要部分。
- **大块修改**：在 Composer 中按方案/计划分步执行，每步可单独验证。

---

## 五、何时开新对话

| 情况 | 建议 |
|------|------|
| 切换新任务、新功能、新模块 | **开新对话**，避免上下文混杂 |
| 同一功能内迭代、修同一处 bug、延续当前实现 | **继续当前对话** |
| Agent 反复犯同类错误、对话已很长且噪音多 | **开新对话**，必要时用 @Past Chats 引用旧结论 |
| 需要旧对话中的结论或计划 | 在当前或新对话中用 **@Past Chats** 引用 |

---

## 六、Commands 与 plans 目录（可选）

- **`.cursor/commands/`**：若有固定、重复的流程，可用 Cursor Commands（`/` 触发）封装，本规则不强制。
- **`.cursor/plans/`**：Plan 模式产出的计划可存于此；与 `work/` 约定并行时，优先使用 `work/YYYY-MM-DD-任务名-计划.md` 便于与工作日志统一管理。
