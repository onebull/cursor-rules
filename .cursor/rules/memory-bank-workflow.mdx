# Memory Bank 工作流规范

> **适用范围**：全仓库任务（分析、设计、实现、验证、总结）。
> **目标**：防止跨会话"断片"；防止重复造轮子；将关键决策显性化并可追溯。

---

## 一、Memory Bank 位置与文件

| 文件 | 职责 |
|------|------|
| `.cursor/memory/activeContext.md` | 当前任务、最近问题、阻塞点、下一步计划 |
| `.cursor/memory/decisionLog.md` | 关键决策（Problem → Solution → Location → Impact） |
| `.cursor/memory/productContext.md` | 项目目标、目标用户、核心流程（可选，首次使用时建立） |
| `.cursor/tasks.md` | 任务进度清单（统一入口，不新增独立 progress 文件） |

> 目录路径可按项目约定调整（如 `project/.cursor/memory/`），但文件结构保持一致。

---

## 二、强制工作流

### 2.1 Pre-action Check（任务开始前必查）

在开始**任何任务**前，必须先读（缺一不可）：

1. `.cursor/memory/activeContext.md`
2. `.cursor/memory/decisionLog.md`
3. `.cursor/tasks.md`

**目的**：了解当前上下文、避免与历史决策冲突、确认任务优先级。

### 2.2 Memory Search（实现前必检索）

在新增工具函数、通用逻辑、流程编排前，必须先检索：

- 代码库现有实现（避免重复实现）
- `decisionLog.md` 既有决策（避免与历史决策冲突）

若已有可用方案，必须「复用或扩展」，不得无理由重写。

### 2.3 Post-action Update（任务完成后必更新）

每个原子任务完成后，同步更新：

1. **`activeContext.md`**：更新当前状态、阻塞点、下一步。
2. **`.cursor/tasks.md`**：更新任务状态（To Do → In Progress → Done）。
3. **`decisionLog.md`**：出现新决策、复杂问题、关键取舍时追加记录。

### 2.4 新增脚本与测试登记（推荐）

新增**可复用脚本**或**关键测试**时，在 `tasks.md` 当次任务下注明「新增：路径 + 用途」，或在对应文档中登记，便于后续检索与复用。

## 三、decisionLog 记录标准

每条记录至少包含：

```markdown
## [标题简述]

- **Date(UTC)**：YYYY-MM-DD
- **Scope**：子项目 / 模块
- **Problem**：遇到了什么问题或需要做什么决策
- **Decision**：最终选择了什么方案，为什么
- **Location**：涉及文件路径
- **Impact**：影响范围 / 回滚提示
```

```markdown
- [已完成的原子任务列表]

- [阻塞点或待用户确认的问题]

## 下一步计划
1. [下一个原子任务]
2. ...
```

## 五、与文档体系的边界

- **权威规范**：仍以 `docs/specs/`、`docs/architecture/`、规则文件为准。
- **工作日志**（`work/`）：历史记录与任务总结归档，引用时标注「历史记录，可能过期」。
- **Memory Bank**：「执行态上下文 + 决策索引」，不替代权威文档。三者互补，不重复。

## 六、项目扩展提示

> 以下为可选扩展方向，项目可按需在此文件追加或新建子规则。

```
# 可扩展维度（按需选用）
#
# graph-memory（关系记忆）
#   - 跨模块影响的实体/关系图谱
#   - 写入触发条件：关键决策 / 跨模块影响 / 易被「与 XX 相关」问到
#   - 格式：实体 + 关系 + observations（摘要+文档路径）
#
# memory-retrieval-pipeline（检索流水线）
#   - 先检索再实现的具体步骤
#   - 引用要求：命中时如何摘要引用
#
# memory-governance（治理规范）
#   - 写入触发条件定义
#   - 冲突优先级
#   - 生命周期与过期清理策略
```
